# PRINCIPLE.md — 코드 규칙

> 이 문서는 Modo 프로젝트의 모든 코드가 따라야 하는 **불변의 원칙**을 정의한다.
> 코드를 작성·수정하기 전에 반드시 이 문서를 먼저 확인할 것.

---

## 0. 핵심 철학 — 로봇 구조 (Robot Architecture)

> **"팔이 이상하면 팔만 고치면 되는 구조"**

로봇은 머리·몸통·팔·다리가 각자 독립된 모듈이다. 한쪽 팔이 고장 나도 다리·몸통은 영향을 받지 않으며, 팔 하나만 분리·수리·교체할 수 있다. 우리 코드도 같다.

- 모든 기능은 **하나의 책임을 가진 객체/모듈**로 분리한다.
- 한 모듈을 수정해도 **다른 모듈에 영향이 없도록** 경계를 명확히 한다.
- 모듈 간 통신은 **약속된 인터페이스(메서드/이벤트)** 를 통해서만 한다.
- 한 곳을 고치기 위해 두 곳 이상을 동시에 고쳐야 한다면 그 설계는 잘못된 것이다 — 즉시 분리하라.

이 원칙을 어기는 코드는 머지될 수 없다.

---

## 1. 모듈화·객체화 규칙

### 1-1. 단일 책임 원칙 (Single Responsibility)
- 하나의 클래스/함수/파일은 **하나의 일**만 한다.
- "그리고(and)"가 설명에 들어가면 분리 신호다.
  - ❌ "체크 상태를 저장하고 UI를 업데이트하는 함수"
  - ✅ "체크 상태를 저장하는 함수" + "UI를 업데이트하는 함수"

### 1-2. 책임의 분리 (Separation of Concerns)
다음 4개의 관심사는 **서로 섞이지 않는다.**

| 계층 | 책임 | 예시 |
|------|------|------|
| **Data** | 데이터 정의·검증 | 영양제 목록, 시간 블록 정의 |
| **Storage** | 영속화·로드 | localStorage 읽기/쓰기, 날짜 비교 |
| **UI / Render** | DOM 생성·갱신 | 카드 그리기, 체크 표시 토글 |
| **Controller / Event** | 사용자 입력 처리 | 클릭 핸들러, 라우팅 |

각 계층은 **자신보다 안쪽 계층만** 호출할 수 있다.
`UI → Storage`, `Controller → UI/Storage` 는 OK. `Storage → UI`는 절대 금지.

### 1-3. 결합도 최소화
- 모듈은 다른 모듈의 **내부 구현**을 알면 안 된다. 공개된 메서드만 사용한다.
- 전역 변수는 사용하지 않는다. 상태는 명시적으로 주입(parameter)하거나 모듈 내부에 캡슐화한다.
- DOM 직접 접근(`document.getElementById`)은 **UI 모듈 내부에서만** 허용된다.

### 1-4. 객체화의 형태
JavaScript에서 객체화는 다음 중 하나의 형태를 따른다 — 상황에 맞게 선택하되, 한 프로젝트 안에서는 일관되게 사용한다.

```js
// (a) ES Module + Class : 상태가 있는 도메인 객체
export class SupplementStorage { ... }

// (b) ES Module + Pure Function : 상태가 없는 유틸
export function isSameDay(a, b) { ... }

// (c) IIFE + Closure : 모듈 시스템 없을 때의 캡슐화
const Storage = (() => { /* private */ return { /* public */ }; })();
```

전역 스코프를 오염시키는 `var Foo = ...` 형태는 금지한다.

## 2. 주석 규칙 (필수)

> 모든 함수·클래스·객체에는 **반드시** 주석을 단다. "당연한 코드"라는 이유로 생략하지 않는다.

### 2-1. 주석에 반드시 포함되어야 할 3가지

| 항목 | 의미 | 질문 |
|------|------|------|
| **왜 (Why)** | 이 코드가 존재하는 이유 | 왜 만들었나? 어떤 문제를 해결하나? |
| **무엇 (What)** | 이 코드가 하는 일 | 무엇을 입력받아 무엇을 출력하나? |
| **상관관계 (Relations)** | 다른 모듈과의 연결 | 누가 이걸 부르나? 이게 누구를 부르나? 무엇에 의존하나? |

### 2-2. JSDoc 표준 양식

```js
/**
 * 체크 상태를 localStorage에 저장한다.
 *
 * 왜:
 *   - 새로고침·앱 재시작 후에도 오늘 체크한 영양제 상태를 유지하기 위해.
 *   - 모바일 브라우저는 백그라운드 진입 시 페이지를 종료할 수 있으므로 즉시 영속화 필요.
 *
 * 무엇:
 *   - 인자로 받은 상태 객체 { [cardId]: boolean } 를 JSON으로 직렬화해 저장.
 *   - localStorage 접근 실패(시크릿 모드 등) 시 조용히 실패 — 앱 크래시 방지.
 *
 * 상관관계:
 *   - 호출처: CheckController.toggle() — 체크박스 클릭 시 즉시 호출됨
 *   - 의존: 브라우저 localStorage API
 *   - 짝: SupplementStorage.load() — 시작 시 같은 키로 상태를 복원
 *
 * @param {Object<string, boolean>} state 카드 ID → 체크 여부
 * @returns {void}
 */
function save(state) { ... }
```

### 2-3. 클래스/모듈 레벨 주석

```js
/**
 * SupplementStorage
 *
 * 왜:
 *   체크 상태의 영속화/복원/일일 초기화 로직을 한 곳에 모아,
 *   UI나 컨트롤러가 localStorage 키 이름·날짜 비교를 알지 못하도록 격리한다.
 *
 * 무엇 (공개 API):
 *   - save(state)         : 현재 체크 상태 저장
 *   - load()              : 저장된 체크 상태 반환 (없으면 빈 객체)
 *   - resetIfNewDay()     : 마지막 저장일과 오늘이 다르면 자동 초기화
 *   - clear()             : 수동 초기화
 *
 * 상관관계:
 *   - 위(상위 호출자): App.init(), CheckController
 *   - 아래(하위 의존): 브라우저 localStorage
 *   - 절대 의존하지 않음: DOM, UI 모듈
 */
class SupplementStorage { ... }
```

### 2-4. 인라인 주석은 "왜"만
한 줄짜리 인라인 주석은 코드가 **무엇을** 하는지가 아니라 **왜** 그렇게 했는지를 설명한다.

```js
// ❌ 나쁨 — 코드를 읽으면 알 수 있는 내용
i++; // i를 1 증가

// ✅ 좋음 — 코드만 봐서는 알 수 없는 의도
btn.classList.toggle('checked'); // 체크 직후 saveState() 호출 전에 시각 피드백을 먼저 — 클릭 지연감 제거
```

### 2-5. 주석을 생략해도 되는 경우
- 일회성 스크립트, 테스트 픽스처
- 공개 API가 아닌 내부 헬퍼 중 **자기 설명적인 이름**을 가진 것 — 단, 모듈 레벨 주석은 여전히 필수

> 의심스러우면 적는다. 주석을 적기 귀찮을 만큼 큰 함수라면, 그 함수가 너무 크다는 신호다.

## 3. 명명 규칙

| 대상 | 규칙 | 예 |
|------|------|------|
| 파일 (모듈) | kebab-case | `supplement-storage.js` |
| 클래스 | PascalCase | `SupplementStorage` |
| 함수·변수 | camelCase | `saveState`, `isCheckedToday` |
| 상수 | UPPER_SNAKE_CASE | `CACHE_VERSION`, `STORAGE_KEY` |
| boolean | `is·has·can·should` 접두사 | `isChecked`, `hasMidnightPassed` |
| DOM 핸들러 | `on` + 이벤트 | `onCheckClick`, `onResetClick` |
| CSS 클래스 | kebab-case + BEM 또는 의미 명사 | `.card`, `.card--checked`, `.time-block` |

이름만 봐도 **무엇이고 어디 소속인지** 알 수 있어야 한다.

## 4. 변경의 원칙

### 4-1. 추가는 자유, 수정은 신중
- 새 영양제, 새 시간 블록 추가 → **데이터 파일만** 수정 (UI/Storage 코드는 손대지 않는다)
- 데이터를 추가하기 위해 UI 코드를 고쳐야 한다면 그건 데이터-UI 분리가 안 된 것 — 즉시 리팩터링한다.

### 4-2. 한 PR(작업 단위)에는 한 가지만
- 버그 수정 PR에 리팩터링 섞지 않기
- 기능 추가 PR에 스타일 변경 섞지 않기

### 4-3. 백워드 호환 더미 코드 금지
- 사용하지 않는 변수·함수·`// 제거됨` 주석은 즉시 삭제한다.
- 미사용 코드는 git history가 보존하므로 코드베이스에 남길 이유가 없다.

### 4-4. 변경 시 영향 범위 확인 의무
코드를 수정하기 전 **반드시** [STRUCTURE.md](STRUCTURE.md)에서 해당 모듈의 호출처·의존성을 확인한다. 모르고 고치면 다른 곳이 깨진다.

## 5. 안티패턴 (금지)

다음은 발견 즉시 리팩터링 대상이다.

| 안티패턴 | 왜 금지 |
|----------|---------|
| **God Function** — 한 함수가 데이터 로드 + UI 렌더 + 저장까지 다 함 | 단일 책임 위반. 한 곳 고치면 여러 곳 깨짐 |
| **DOM ID 하드코딩 산재** — 여러 모듈이 `getElementById('card-iron')`를 직접 호출 | UI 변경 시 모든 모듈 수정 필요. UI 모듈에 캡슐화할 것 |
| **전역 변수 상태** — `window.checkedItems = ...` | 어디서 누가 변경했는지 추적 불가. 모듈 내부에 캡슐화 |
| **인라인 이벤트 속성** — `<div onclick="toggleCheck(...)">` | HTML과 JS 결합. JS에서 `addEventListener`로 위임 |
| **매직 넘버/문자열** — `if (i === 5)`, `localStorage.getItem('xyz')` | 의미 불명. 상수로 추출 |
| **주석 없는 공개 함수** | 본 문서 §2 위반 |
| **중복 코드** (3회 이상 반복) | DRY 위반. 함수로 추출 |

## 6. 체크리스트 — 코드를 머지하기 전

- [ ] 한 모듈이 **하나의 책임**만 갖는가?
- [ ] 모든 함수·클래스에 **왜·무엇·상관관계** 주석이 있는가?
- [ ] DOM 접근이 **UI 계층 안에만** 있는가?
- [ ] 전역 변수를 만들지 않았는가?
- [ ] 매직 넘버/문자열을 상수로 추출했는가?
- [ ] [STRUCTURE.md](STRUCTURE.md)에 새 파일·관계를 반영했는가?
- [ ] 사용하지 않는 코드를 모두 지웠는가?
- [ ] 한 곳을 고치기 위해 두 곳을 동시에 고치지 않았는가? (로봇 구조 위반 신호)

하나라도 ❌라면 머지 금지. 리팩터링 후 다시 검토한다.
